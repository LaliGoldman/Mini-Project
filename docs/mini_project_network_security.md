## Course format

- A "mini project" in information security / network security.
- One introductory meeting, then independent self-directed work.
- Optional mid-semester meeting(s), arranged by the students (not mandatory).
- Instructor support via email / phone / WhatsApp as needed.

## Course goals (KFD)

- **Know** — understand (even partially) the environment an attacker operates in.
- **Feel** — experience how a hostile intruder thinks and feels.
- **Do** — build the tool.

Sub-goals: understand the attacker's mindset and multi-layered defense; know threat types, defense approaches/mechanisms, and system weak points; be able to formulate and implement a security policy.

---

## The assignment

The cyber world contains a large set of tools and possibilities. The goal is to experience **building / operating** such a tool — or defending against one.

You are required to build, **for example**, a piece of "malware" of your choice:
- A trojan, a virus, or any other kind of malware.
- You must also address the **distribution/propagation mechanism** for that malware. Distributing such things is hard, and building a **creative propagation mechanism** is a challenge in itself.

### Topic selection

- One project chosen from a range of topics.
- Topics relate directly to cyber security and may include: building malware, building an attack platform, building an attack, etc.
- A list of example topics is available on the course site.

### Team size

- The project is **individual**.
- Can be done in **pairs**.
- **Trios require special approval** (per the work plan).
- The grade is individual and also reflects **your** activity/contribution to the project.

## Deliverable 1 — Work Plan (ת"ע / תוכנית עבודה)

Submit **by email**. Submitting the work plan with a schedule is a **prerequisite** to continue.

Must include:
- **Specification document** — system scope/boundaries, description of scenarios.
- **Goal / objectives.**
- **Milestones and schedule.**
- **Challenges and difficulties.**
- **Strengths.**

The work plan is also **attached to the final documentation**.

**Deadline:** up to one month after the semester starts — **15 April 2026**.

## Deliverable 2 — Project + Documentation

Submit the documentation **together with the project**. Must include:
- What the tool does, and whether it meets its goal.
- Problems you encountered.
- Strengths and weaknesses of the tool.
- Description of how the tool works.
- Documentation of the code and its main components (may be documented in the code itself).
- Links/references to any modules, code snippets, or tools you used.

**Recommended timing:** submit at ~80–90% of the semester (before the exam period) to leave time for corrections.

## Grading

Grade is based on the project only:
- Up to **15 pts** — use of **social engineering**.
- Up to **15 pts** — **creativity and breaking paradigms**.
- Up to **15 pts** — **documentation**.
- Up to **5 pts** — submitting the work plan **on time**.
- Remaining points — the **work and the code**.
